# User Flows

Документ описывает ключевые сценарии использования продукта — как
пользователь решает конкретную задачу, проходя через экраны и развилки.
Используется как основа для wireframes и проверки полноты карты экранов.

---

## Flow 1: Репетитор приглашает первого ученика

**Триггер:** Tutor только что зарегистрировался, видит пустой дашборд.
Хочет начать работу с учеником N (с которым уже договорился вне системы).

**Цель:** ученик N зарегистрирован в системе и привязан к репетитору.

### Часть 1: Tutor создаёт карточку и получает ссылку

1. Tutor на `/` (Dashboard, **пустое состояние**, CTA «Добавить ученика»)
2. Клик CTA → `/students/new`
3. Заполняет форму: ФИО, класс, предмет, контакт, заметки
4. Submit:
    - ❌ Невалидные данные (пустое ФИО, неверный класс) → ошибка на форме,
      остаётся на `/students/new`
    - ✅ Ок → запись Student создана (без email/пароля — будут заданы
      при регистрации) → переход на `/students/:id`
5. На `/students/:id` (карточка ученика, **пустая**, без ДЗ)
6. Клик «Пригласить» → открывается **модалка** с ссылкой
   `app.com/invite/abc123` (создан Invite со статусом `ACTIVE`,
   `expires_at = now + 7 дней`)
7. Клик «Скопировать» → ссылка в буфере, toast «Скопировано»
8. Tutor закрывает модалку, отправляет ссылку ученику в мессенджер
   (вне системы)

⊙ **Промежуточный конец:** ссылка отправлена, ждём действий ученика.

### Часть 2: Ученик регистрируется по ссылке

9. Ученик открывает ссылку `app.com/invite/abc123`
    - ❌ Ссылка истекла (`status = EXPIRED` или `expires_at < now`) →
      экран «Ссылка истекла, попросите репетитора прислать новую»
    - ❌ Ссылка уже использована (`status = USED`) →
      экран «Ссылка использована, войдите в свой аккаунт» → CTA на `/login`
    - ✅ Ссылка активна → форма регистрации (email, пароль) на той же странице
10. Ученик заполняет email и пароль, submit:
    - ❌ Email уже занят (есть среди других tutors/students) →
      ошибка на форме
    - ❌ Невалидный пароль (короткий, пустой) → ошибка на форме
    - ✅ Ок → создаётся учётка Student, привязка к существующей карточке
      (`tutorId` уже задан), Invite помечается `USED`, заполняется `usedAt`
11. Автоматический логин → переход на `/` (Student Dashboard)

### Состояние после flow

- **Tutor:** в `/students` у него один ученик. На карточке `/students/:id`
  виден индикатор «зарегистрирован» (можно показывать `createdAt` Student'а).
- **Student:** на `/`, видит пустой дашборд (ДЗ ещё нет),
  empty state «Пока нет заданий — ждите от репетитора».

⊙ **Конец flow.**

### Edge cases (учесть на этапе разработки)

- Tutor может **повторно** открыть карточку и сгенерировать новую ссылку,
  если старая протухла или потерялась. Старая инвалидируется.
- Что если Tutor удалил карточку, пока Invite активен? — При попытке
  регистрации показать «Приглашение недействительно».

## Flow 2: Репетитор выдаёт ДЗ ученику

**Триггер:** в системе уже есть зарегистрированный ученик. Репетитор
хочет выдать ему домашнее задание.

**Цель:** ДЗ создано, видно у репетитора (статус `IN_PROGRESS`)
и появилось у ученика на дашборде.

### Шаги

1. Tutor на `/` (Dashboard, есть ученики, есть/нет ДЗ — неважно)
2. Клик «Ученики» в сайдбаре → `/students`
3. Видит список учеников, кликает на нужного → `/students/:id`
4. На карточке ученика: его инфо + список ДЗ (может быть пустым)
5. Клик «Создать ДЗ» → `/students/:id/assignments/new`
6. Заполняет форму:
    - Название (обязательно)
    - Описание (обязательно, длинный текст)
    - Дедлайн (обязательно, должен быть в будущем)
    - Прикрепить файлы (опционально, например PDF с задачами)
7. Submit:
    - ❌ Невалидные поля (пустое название, дедлайн в прошлом) →
      ошибка на форме
    - ❌ Файл слишком большой / неподдерживаемый формат → ошибка
    - ✅ Ок → запись Assignment создана (`status = IN_PROGRESS`,
      `tutorId`, `studentId` заполнены), файлы загружены в хранилище →
      переход на `/assignments/:id`
8. На `/assignments/:id` Tutor видит созданное ДЗ:
    - Все введённые данные
    - Прикреплённые файлы
    - Статус «В работе»
    - Блок «Ответ ученика» — пусто, плейсхолдер «Ответа пока нет»

⊙ **Конец flow со стороны Tutor.**

### Что произошло параллельно у ученика

- На `/` (Student Dashboard) появилось новое ДЗ в блоке «Активные»
- На `/assignments` — добавилось в список со статусом «В работе»
- Если бы у ученика были включены email-уведомления — пришло бы письмо
  *(не в MVP, упоминаем как задел)*

### Состояние после flow

- **Tutor:** на `/students/:id` в списке ДЗ ученика появилось новое
  с статусом «В работе». На дашборде это ДЗ **не отображается**
  в блоке «Требует проверки» (потому что не сдано).
- **Student:** на `/` видит ДЗ в блоке «Активные» с дедлайном.

### Edge cases

- Дедлайн через 5 минут — формально валидно, но странно. Решение:
  валидация только «дедлайн в будущем», нижнего порога нет.
- Tutor создал ДЗ и сразу удалил — Student мог увидеть его, если успел
  обновить страницу. Решение для MVP: считаем редким случаем, не обрабатываем.

## Flow 3: Ученик сдаёт ДЗ и репетитор проверяет

**Триггер:** Ученик решил задачу вне системы (на бумаге, в тетради).
Открывает приложение, чтобы отправить ответ репетитору.

**Цель:** ДЗ переходит в статус DONE — репетитор принял работу.

---

### Часть 1: Ученик отправляет ответ

1. Student на `/` (Dashboard) или `/assignments`. Видит активное ДЗ
   в блоке «Активные» / в списке.
    - Если дедлайн уже прошёл, а статус ещё IN_PROGRESS → ДЗ помечено
      красным «Просрочено», но кнопка «Сдать» **остаётся доступной**
      (решение для MVP: разрешаем сдавать с опозданием).
2. Клик по ДЗ → `/assignments/:id`
3. На странице видит:
    - Описание задачи, файлы от репетитора, дедлайн, статус «В работе»
    - Форму отправки ответа: поле «Текст ответа» + кнопка «Прикрепить файлы»
4. Заполняет форму:
    - Пишет текст и/или прикрепляет файлы (хотя бы что-то одно — обязательно)
5. Клик «Отправить ответ»:
    - ❌ Пустой ответ (нет текста и нет файлов) → ошибка валидации,
      остаётся на странице
    - ❌ Файл слишком большой / неподдерживаемый формат → ошибка
    - ✅ Ок → создаётся Submission (text + связанные File),
      статус Assignment: `IN_PROGRESS → UNDER_REVIEW`
6. После отправки страница перезагружается / обновляется:
    - Форма скрыта, вместо неё — блок «Ваш ответ» с текстом и файлами
    - Статус «На проверке»
    - Toast «Ответ отправлен, ждите проверки»

⊙ **Промежуточный конец:** ответ отправлен, ждём действий репетитора.

---

### Часть 2: Репетитор проверяет работу

7. Tutor на `/` (Dashboard). В блоке **«Требует проверки»** появилось
   это ДЗ (потому что статус UNDER_REVIEW).
    - Альтернативный путь: Tutor может зайти через `/students` →
      карточку ученика → ДЗ. Но дашборд — основной сценарий.
8. Клик по ДЗ → `/assignments/:id`
9. На странице видит:
    - Описание ДЗ (его собственное)
    - Файлы, которые он прикреплял
    - **Блок «Ответ ученика»**: текст ответа + файлы от ученика
    - Дата отправки ответа
    - Две кнопки: «Принять работу» и «Вернуть на доработку»
10. Tutor выбирает действие:
    - ✅ **Принять** → PATCH-запрос, статус `UNDER_REVIEW → DONE`,
      toast «Работа принята»
    - ↩️ **Вернуть на доработку** → PATCH-запрос,
      статус `UNDER_REVIEW → IN_PROGRESS`, **Submission удаляется**
      (продуктовое решение для MVP: ученик заново отправляет ответ).
      Toast «Работа возвращена ученику»
11. Страница обновляется, статус сменился:
    - Если DONE — кнопки скрыты, блок «Ответ ученика» остаётся для истории
    - Если вернули — блок «Ответ ученика» исчезает, ДЗ опять
      выглядит как новое для ученика

---

### Состояние после flow (сценарий «Принято»)

- **Tutor:** на `/` ДЗ исчезло из «Требует проверки»,
  на карточке ученика статус ДЗ — «Сделано». Если в дашборде есть
  счётчик «всего проверено» — увеличился на 1.
- **Student:** на `/` ДЗ исчезло из «Активные», на `/assignments` —
  во вкладке «Сделанные». Toast/уведомление в системе:
  «Репетитор принял ваш ответ» *(не в MVP, на будущее)*.

### Состояние после flow (сценарий «Вернуть»)

- **Tutor:** на `/` ДЗ исчезло из «Требует проверки»
  (статус IN_PROGRESS), но осталось в карточке ученика.
- **Student:** ДЗ снова в «Активные», статус «В работе», форма
  отправки ответа снова доступна, прежний ответ не сохранён.

### Edge cases

- **Просроченное ДЗ:** статус IN_PROGRESS + дедлайн < now.
  Решение: разрешаем сдавать, помечаем красным.
- **Возврат на доработку:** удаляем Submission. Альтернатива —
  оставлять историю попыток (отложено на v2).
- **Tutor удалил ДЗ, пока ученик пишет ответ:** при попытке отправить
  ответ — 404 / «ДЗ не найдено». Для MVP не обрабатываем спец. UI.
- **Ученик нажал submit дважды быстро:** на бэке защита от дублей
  (один Submission на одно Assignment, поле `unique`). Frontend —
  блокирует кнопку после первого клика.
